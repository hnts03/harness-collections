# harness 지시사항
> **🔧 `/harness` 전용 — `/pm`은 이 섹션을 실행하지 않는다.**
> 하네스 빌딩 시 1회 실행. 팀 구조·에이전트·스킬을 설계하고 생성하는 단계.

## 팀 빌딩 원칙

- 하네스는 reusable 해야 합니다. 처음에 팀 빌딩에 제시된 요구가 주어지지 않더라도 robust하게 작업을 할 수 있어야 합니다. 유저가 특정한 주제를 주었다 하더라도 그 업무에 편향되지 않도록 해야합니다.
- 하네스 구성의 개괄은 `/harness` 스킬 트리거 할 때 제공하나, 디테일한 매 작업 목표(이슈)는 `/pm` 스킬로 PM 에이전트를 트리거해서 유저가 직접 제공합니다. 그것을 바탕으로 작업 계획부터 실행, 리뷰, 테스트 자동화, QA, 문서화 프로세스가 완전히 구성될 수 있어야 합니다.
- 전문적으로 계획을 구성하고 태스크를 생성·관리할 수 있도록 PM 에이전트가 `/pm` 스킬로 트리거되어 실행되어야 합니다. 계획이 필요한 상황이라면 claude-code의 planner agent를 사용하여도 좋습니다.
- 테스트 자동화 프로세스까지 포함되어야 하고, 동일한 결과를 유저가 재생산할 수 있도록 가이드(WORKGUIDE)가 있어야 합니다.
- 팀 빌딩이 완료되면 **프로젝트 루트의 `WORKGUIDE.md`** 에 빌드된 팀의 사용법을 정리해두세요. (`docs/`는 생성된 산출물 디렉토리로 gitignore 처리되므로 WORKGUIDE는 루트에 위치해야 합니다.)
- 팀 빌딩 완료 시 **`CLAUDE.md`를 생성**한다. 팀 구성(에이전트·모델·역할), 스킬 목록, 운영 원칙을 기록하고 `## 완료된 세션 이력` 섹션을 빈 상태로 추가한다. 이후 `/update-from-phase` 스킬이 이 섹션에 세션 이력을 누적한다.

## Phase 0: 도메인 분석

팀 빌딩 전에 반드시 도메인 분석을 먼저 수행한다. 분석 결과는 패턴 선택과 역할군 구성의 직접적인 입력이 된다.

1. **도메인/프로젝트 파악** — 유저 요청에서 핵심 목적과 주요 작업 유형(생성, 검증, 편집, 분석 등)을 식별한다.
2. **재사용성 분석** — 이 팀이 반복적으로 처리하게 될 작업 유형을 파악한다. 특정 이슈에 편향되지 않는 범용 팀을 구성하기 위해 도메인의 공통 패턴을 추출한다.
3. **기존 에이전트/스킬 확인** — `.claude/agents/`와 `.claude/skills/`를 탐색하여 충돌·중복을 방지한다.
4. **코드베이스 탐색** (프로젝트가 있는 경우) — 기술 스택, 데이터 모델, 주요 모듈을 파악한다.
5. **분석 요약 도출** — 위 결과를 한 페이지로 요약한다. 이 요약이 다음 단계(패턴 선택, 역할군 구성)의 근거가 된다.

> **[체크포인트 1/4 — 도메인 분석 완료]**
> 핵심 작업 유형·재사용 패턴·기존 에이전트/스킬 현황을 한 페이지 요약으로 유저에게 제시한다. 확인을 받은 뒤 패턴 선택으로 넘어간다.

## 아키텍처 패턴 선택

도메인 분석 후 아래 기준으로 팀 아키텍처 패턴을 선택한다. **실전에서는 단일 패턴보다 복합 패턴이 흔하므로 조합도 적극 고려한다.**

| 패턴 | 선택 기준 |
|------|----------|
| **파이프라인** | 각 단계가 이전 단계 산출물에 강하게 의존하는 순차 작업 |
| **팬아웃/팬인** | 동일 입력에 대해 서로 다른 관점의 병렬 분석이 필요한 경우 |
| **전문가 풀** | 입력 유형에 따라 다른 전문가를 선택적으로 호출해야 하는 경우 |
| **생성-검증** | 산출물 품질 보장이 중요하고 객관적 검증 기준이 존재하는 경우 |
| **감독자** | 작업량이 가변적이거나 런타임에 작업 분배를 동적으로 결정해야 하는 경우 |
| **계층적 위임** | 문제가 자연스럽게 계층적으로 분해되는 구조 (최대 2단계 권장) |

**에이전트 팀 vs 서브 에이전트:**
- 2개 이상 에이전트가 협업하면 **에이전트 팀**이 기본값
- 에이전트 간 통신이 불필요하거나 단일 에이전트면 서브 에이전트

> 상세 기준·팀 모드 적합성·복합 패턴 예시는 `references/agent-design-patterns.md` 참조.

> **[체크포인트 2/4 — 아키텍처 패턴 확정]**
> 선택한 패턴(단일 또는 복합)과 선택 이유를 유저에게 제시한다. 패턴이 잘못되면 이후 역할군 구성 전체가 틀어지므로, 확인을 받은 뒤 에이전트/스킬 정의로 넘어간다.

## 역할군 및 스킬 구성 최소 만족 조건

하네스 빌딩 시 아래 조건은 반드시 만족해야한다. 그 외에 팀 운영 시 필요하다고 판단되는 경우 스킬과 에이전트를 직접 만들어서 사용한다.

### 에이전트

모든 agent는 각 분야에 대해 20년 이상의 경력자들이다.

- **project-manager** : 주어진 미션에 대응하여 high-level research를 수행하고 그것을 바탕으로 작업 plan 및 작업자들에게 업무를 분배·오케스트레이션 한다. 서브 에이전트들을 병렬화하여 업무를 처리시키며, 각 phase와 task 간 의존성을 사전에 파악하여 최적의 병렬 작업을 달성한다.
  - **[resume detection 필수]** 트리거 시 가장 먼저 아래 순서로 이전 세션 여부를 확인한다:
    1. **플랜 번호 먼저 확정**: `_workspace/` 하위 숫자형 디렉토리 최댓값 + 1을 `NEXT_PLAN_NO`로 고정. 없으면 `001`. (채번 규칙: 아래 PM 운영 지침 참조)
    2. `_workspace/*/*-dev-plan.md` 파일 존재 여부 및 진행 상태 확인
    3. 진행 중인 플랜 발견 → 유저에게 이어서 진행할지 새로 시작할지 확인 후 분기
       - **이어서** → 해당 플랜 번호 유지 (`NEXT_PLAN_NO` 무시)
       - **새로 시작** → `NEXT_PLAN_NO` 사용
    4. 플랜 없음 → `NEXT_PLAN_NO`로 신규 세션 시작
- **architect** (optional) : 개발 업무의 경우 프로젝트 구조, 내부 컴포넌트 간 인터페이스, 외부 API 등 다양한 구조 정의를 사전에 구성한다. **직접 파일을 탐색하지 않고 worker를 fan-out 스폰하여 데이터를 수집**한 뒤, 취합된 결과만 보고 설계에 집중한다 (탐색 위임 패턴).
- **worker** : PM이 만들어둔 plan의 task를 실제 수행하는 에이전트로 개발·탐색 등 일반적인 업무를 담당한다.
  - **[Scope 자가진단 필수]** task 수신 즉시 아래 조건을 확인한다. 하나라도 해당하면 구현 미착수 상태로 `failed` 보고 후 분해 제안:
    1. `outputs.files` 수가 4개 이상이거나 건드릴 파일이 3개 초과로 예상될 때
    2. description에 복수의 독립적 concern이 혼재할 때 (예: "파싱 + 저장 + 알림")
    3. acceptance_criteria가 단일 task로 검증 불가할 만큼 넓을 때
- **reviewer** : worker의 작업을 리뷰한다. worker role이 개발이면 코드 리뷰, 탐색이면 정보 더블체크를 수행한다.
- **qa** : phase 단위의 QA를 수행한다. QA 완료 후 결과 보고서를 생성하여 PM이 확인하도록 한다.
- **document-writer** : 미션이 완료되었을 때 결과를 정리하는 문서를 생산한다.

### 스킬

- **pm** : project-manager 에이전트를 트리거하는 진입점 스킬. `/pm` 명령으로 호출하며, resume detection → 미션 수령 → 플랜 수립 → 오케스트레이션 전 과정을 시작한다.
- **create-skill** (optional) : 필요한 skill을 생성하는 skill이다. 이미 유사 skill이 존재한다면 생성하지 않아도 된다. 스킬 명세 파일명은 반드시 `SKILL.md`로 고정한다.
- **create-agent** (optional) : 필요한 agent를 생성하는 skill이다. 이미 유사 agent가 존재한다면 생성하지 않아도 된다.
- **clean-commit** : claude-code는 git commit 시 co-author 문구를 삽입한다. 이 문구를 제거하고 commit하는 기능이다.
- **update-from-phase** : phase 완료 시 PM이 직접 트리거한다. dev-plan.md의 phase 상태 마킹, CLAUDE.md·memory 갱신, git commit & push를 일괄 처리한다. task 단위 갱신은 PM이 직접 수행하며 이 스킬은 덮어쓰지 않는다.

> **[체크포인트 3/4 — 팀 구성 확정]**
> 에이전트 목록(역할·모델), 스킬 목록을 유저에게 제시한다. 확인을 받은 뒤 실제 에이전트/스킬 파일 생성으로 넘어간다.

## harness 동작 재정의

- **SKILL.md 파일명 강제**: 현재 harness 스킬의 버그로 인해 skill creation 시 `skill.md`로 작성하는 버그가 있습니다. skill을 만들 때 반드시 `SKILL.md` 파일명으로 작업하도록 강제하세요.
- **모델 선택 기준**: 현재 harness 스킬은 모든 agent를 `opus`로 생성하지만, 역할 복잡도에 따라 아래 기준으로 모델을 선택한다.

  | 모델 | 역할 | 이유 |
  |------|------|------|
  | **opus** | project-manager | 전체 계획 수립·의존성 분석·오케스트레이션 결정 등 복잡한 판단 |
  | **opus** | architect | 시스템 구조 설계·인터페이스 정의 등 높은 추론 복잡도 |
  | **opus** | reviewer | 품질 판단·미묘한 결함 감지 등 섬세한 평가 능력 필요 |
  | **opus** | qa | 경계면 교차 비교·통합 정합성 검증 등 정확성이 핵심 |
  | **sonnet** | worker | 명확히 정의된 task 실행 중심. 비용 절감 효과 가장 큼 |
  | **sonnet** | document-writer | 구조화된 문서 작성으로 복잡한 추론 불필요 |

  > **예외**: worker라도 요구사항이 모호하거나 아키텍처 수준의 판단이 필요한 task는 `opus`로 업그레이드한다. 모델 선택은 역할 고정이 아니라 **task 복잡도 기반**임을 기억한다.

- **Worker 범위 강제 규칙**: PM은 worker 스폰 전에 아래를 반드시 검토한다.
  - 수정 파일 3개 초과 → subtask 분해 후 각각 별도 worker 스폰
  - concern 복수 → concern별 분리
  - 병렬 스폰 시 `outputs.files` 교집합 확인 → 겹치면 순차 실행

# PM 운영 지침
> **🚀 `/pm` 전용 — 하네스 팀이 구성된 후 PM 에이전트가 매 세션(이슈)마다 따르는 운영 규칙.**
> `/pm`이 이 파일을 받으면 이 섹션만 참조한다. 위의 "harness 지시사항" 섹션은 무시한다.

## 세션 시작

- 트리거 시 반드시 **resume detection**을 먼저 수행한다. 순서:
  1. `NEXT_PLAN_NO` 채번 (아래 규칙 참조)
  2. `CLAUDE.md` 읽기 → `## 완료된 세션 이력`으로 이전 이슈 맥락 파악. 의존성 있는 이슈라면 이 정보를 미션 명확화에 활용한다.
  3. `_workspace/*/*-dev-plan.md` 확인 → 진행 중 플랜 있으면 유저에게 이어서/새로 시작 여부 확인
- **플랜 넘버 채번 규칙 (resume detection 최우선 단계)**:
  - `_workspace/` 하위 숫자형 디렉토리의 최댓값 + 1. 디렉토리가 없으면 `001`. 형식: 3자리 제로패딩.
  - 예: `_workspace/001/`, `_workspace/002/` 존재 → 다음 플랜 넘버 `003`.
  - **이 값(`NEXT_PLAN_NO`)을 먼저 확정한 뒤**, 기존 플랜 파일(`*-dev-plan.md`) 검사를 수행한다. 어떤 분기를 타든 플랜 번호가 `001`로 초기화되지 않도록 한다.
- 작업 시작 전 `<플랜 넘버>-dev-plan.md` 파일에 계획과 현재 진행상황을 기록한다. 플랜 넘버는 하네스 팀이 재사용될 때 각 세션의 플랜을 구별하기 위한 식별자다.
  - **`dev-plan.md` 구조** — 아래 구조를 따른다. 이 파일은 PM이 task마다 직접 갱신하는 live document다.

    ```markdown
    ## 목표
    [세션의 목표]

    ## 팀 구성
    - 아키텍처 패턴: [선택된 패턴]
    - 에이전트: [역할 — 모델 목록]

    ## Phase 계획 및 현황

    ### Phase 1: [이름]
    **상태**: 대기 / 진행 중 / 완료
    **목표**: [phase 목표]
    **담당**: [에이전트]

    #### Tasks
    - [ ] task 1 — [담당] — 대기
    - [~] task 2 — [담당] — 진행 중
    - [x] task 3 — [담당] — 완료

    ### Phase 2: [이름]
    ...

    ## 현재 상태
    - **진행 중인 Phase**: Phase N
    - **진행 중인 Task**: [task 설명]
    - **다음 작업**: [다음 task 또는 phase]

    ## 이슈 / 결정 사항
    - [날짜] [내용]
    ```
- 팀원 간 소통 및 작업 도중 문서화가 필요한 경우 `_workspace/<플랜 넘버>/` 디렉토리 안에서 작업공간을 격리하여 관리한다.
- 계획 수립 시 모호한 부분이 있으면 무조건 유저와 align을 맞추고 이어서 진행한다.

> **[체크포인트 4/4 — 플랜 확정]**
> 작성한 `<플랜 넘버>-dev-plan.md`(phase 구성, 의존성, 담당 에이전트)를 유저에게 제시한다. 확인을 받은 뒤 실제 작업 실행으로 넘어간다.

## 작업 실행

- PM이 구성한 plan의 작업은 **phase**(큰 단위)와 그 하위 **task**(세부 단위)로 스케줄된다.
- 상호 의존성 없는 작업들은 병렬로 실행한다. 각 작업 간 의존성 체크를 사전에 수행해야 한다. **병렬 스폰 전 `outputs.files` 겹침을 반드시 확인한다 — 파일이 겹치는 task는 순차 실행.**
- **Task 분해 원칙**: 하나의 task는 단일 concern, 수정 파일 최대 3개를 기준으로 한다. 범위가 넓으면 subtask로 분해하여 각각 별도 worker에게 할당한다. worker를 더 많이 쓰더라도 각 worker의 범위를 좁게 유지한다.
- **작업 도중이라도** 모호한 요구사항·설계 결정·판단 기준이 생기면 즉시 유저에게 질문하여 align을 맞추고 진행한다. 불확실한 채로 진행하지 않는다.
- **task가 완료될 때마다** `dev-plan.md`의 task 체크박스와 `## 현재 상태` 섹션을 즉시 갱신한다. 이 갱신이 누락되면 resume detection이 부정확해진다.

## 실패 처리 정책

**기본 원칙**: 재시도는 최대 1회. 재시도 후 동일 실패 시 즉시 에스컬레이션하며 무한 루프를 허용하지 않는다. 모든 실패 내용은 `_workspace/<플랜 넘버>/failures.md`에 기록한다.

| 실패 유형 | 1차 대응 | 2차 대응 (재시도 후도 실패) |
|----------|---------|--------------------------|
| **worker 실패** | worker가 원인 분석 후 1회 재시도 | PM에게 에스컬레이션 |
| **reviewer reject** | worker가 피드백 반영 후 1회 재작업 | PM에게 에스컬레이션 |
| **QA fail** | 해당 task 담당 worker가 1회 수정 후 재검증 | PM에게 에스컬레이션 |

**PM 에스컬레이션 시 PM의 판단:**
1. PM이 대안 접근법을 직접 시도할 수 있으면 → 시도 (1회 한정)
2. 요구사항 명확화가 필요하거나 PM도 해결 불가 판단 시 → 유저에게 에스컬레이션

**유저 에스컬레이션 시 필수 포함 내용:**
- 무엇이 실패했는가 (실패 내용과 에러)
- 무엇을 시도했는가 (시도한 접근법)
- 무엇이 필요한가 (유저에게 요청하는 결정 또는 정보)

## Phase 완료 시

- 각 phase 완료마다 `docs/phase_<N>/` 디렉토리에 프로세스 진행 내용을 설명하는 마크다운 문서를 작성한다.
- **`update-from-phase` 스킬을 트리거**하여 아래를 일괄 처리한다:
  - `dev-plan.md`: 해당 phase를 `완료`로 마킹, 다음 phase를 `진행 중`으로 전환
  - CLAUDE.md·memory 갱신
  - git commit(co-author 문구 제거) & push (연결된 remote repo가 있으면)
  > PM이 task마다 갱신한 dev-plan.md 내용은 유지하고, phase 상태만 덮어쓴다.

## 세션 종료

- 모든 작업 완료 후 `/clear`로 컨텍스트를 초기화하고 다음 작업을 준비한다.
- 유저가 별도 요구하지 않더라도 종료 시점에 CLAUDE.md·memory·관련 문서에서 현재까지의 context를 확인할 수 있어야 한다.
